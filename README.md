# Zenlit

Zenlit is a web-first, native-ready Expo app for location-first networking. The current product target is the browser MVP: mobile Safari/Chrome first, desktop browser second, native apps later only after traction. The main app entry is Radar, with supporting Feed, Create, Chat, and Profile screens. Authentication uses Supabase email OTP, public profile data lives in Supabase, and onboarding is guarded by a backend-driven resolver before the bottom tabs are shown.

## Current Direction

- Build and test the web product first: `npm run dev`, `npm run build:web`, Vercel deployment, and Docker web packaging.
- Keep Expo + React Native Web rather than rewriting to pure web.
- Keep service, validation, auth, onboarding, profile, feed, messaging, and storage logic portable for future native apps.
- Treat iOS/Android builds, app store assets, native push tokens, and native permission work as traction-gated future work.

## Quick Start

Prerequisites:

- Node.js 22.13 or newer and npm
- Expo CLI through `npx expo`
- A Supabase project with email OTP enabled
- Vercel project environment variables or Docker build args for production web deploys

Install dependencies:

```bash
npm install
```

Create a local `.env` file:

```bash
EXPO_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
EXPO_PUBLIC_SUPABASE_ANON_KEY=your-anon-or-publishable-key
```

Only Supabase anon/publishable keys belong in Expo public environment variables. Do not put service-role keys or other secrets in `.env` values that start with `EXPO_PUBLIC_`.

Run the web app:

```bash
npm run dev
```

Build the web app:

```bash
npm run build:web
```

## Docker

Build and run the production Expo Web container:

```bash
docker compose up --build
```

Open `http://localhost:8081`. Compose reads `.env` and passes the `EXPO_PUBLIC_SUPABASE_*` values as build args. The container serves the static `dist/` export with Nginx and falls back to `index.html` for Expo Router routes.

## Web Deployment

Vercel and Docker are both supported web deployment paths. `vercel.json` uses `npm run build:web`, outputs `dist`, and rewrites routes to support browser refreshes on app paths such as `/messages/[id]` and `/profile/[id]`.

Set `EXPO_PUBLIC_SUPABASE_URL` and `EXPO_PUBLIC_SUPABASE_ANON_KEY` in Vercel project environment variables.

Docker uses the same `npm run build:web` export and serves it through Nginx. Pass `EXPO_PUBLIC_SUPABASE_URL` and `EXPO_PUBLIC_SUPABASE_ANON_KEY` through `.env` when running `docker compose up --build`.

## Onboarding Summary

Authenticated routing is intentionally centralized. `src/services/onboardingService.ts` reads Supabase profile state and resolves route targets, while `src/utils/onboardingState.ts` evaluates local onboarding state used by the root layout and onboarding screens.

The required journey is:

```text
Get Started -> Email OTP -> Profile Basics -> Optional Profile Details -> Radar
```

Required profile basics are display name, username, date of birth, and gender. Optional details are avatar, banner, bio, Instagram, X, and LinkedIn. Complete Profile is shown once after basics; Save or Skip unlocks Radar.

## Supabase Summary

The connected remote public schema currently exposes:

- Tables: `profiles`, `profile_basics_drafts`, `social_links`, `posts`, `feedback`, `locations`, `messages`
- View: `locations_current`
- RPCs: chat authorization, email/username availability, username suggestions, location expiry, unread counts, and message delivery/read markers
- Storage buckets from local migrations and app usage: `profile-images`, `post-images`, `feedback-images`
- Edge functions in source: `send-push-notification`, `update-conversation-anonymity`

Read-only checks on the active remote project confirmed `profile_basics_drafts`, profile draft RLS policies, and `profiles.optional_profile_completed_at` are present.

## Testing

The test foundation uses Jest with `jest-expo` and React Native Testing Library. Tests live outside `app/`:

- `test/unit/` for pure routing, validation, and parsing logic
- `test/ui/` for screen/component/router behavior with mocked native modules
- `test/setup/jest.setup.tsx` for native, Expo, and Supabase test doubles
- `test/utils/render.tsx` for shared render helpers

Useful commands:

```bash
npm run typecheck
npm run test:unit
npm run test:ui
npm test
npm run build
```

More details are in `docs/knowledge-base/testing.md`.

## Common Development Workflow

1. Pull the latest branch and run `npm install`.
2. Add or update `.env` with local Supabase public values.
3. Make scoped app changes in `app/`, `src/`, or `supabase/`.
4. Add unit tests for pure logic and UI tests for high-risk screens.
5. Run `npm run typecheck`, targeted tests, then `npm run verify` before handing off.
6. If backend structure changes, update `supabase/migrations/`, generated types, and `docs/knowledge-base/backend.md` in the same task.
